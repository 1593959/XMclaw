## 一、架构总览

XMclaw 记忆系统采用 **V1（legacy）+ V2（MemoryService）双轨并存** 架构，V2 为当前主力：

| 层级 | 职责 | 关键文件 |
|------|------|----------|
| **L0 事件层** | 原始对话 turn、工具调用、文件变更等不可变事件 | `events.db`（审计日志） |
| **L1 事实层** | 结构化 Fact（偏好/身份/决定/项目/教训等）+ Relation 图 | `memory.db`（LanceDB / InMemory） |
| **L2 主题层** | LLM 聚类命名的事实群组（topic nodes） | 由 `llm_topic.py` 动态生成 |
| **L3 自传层** | 长会话摘要、跨会话叙事连续性 | `autobiographical_memory.py` |

**后端选择**：
- **生产环境**：LanceDB（向量 + 图混合存储）
- **测试环境**：InMemory（Dict-backed，无持久化）
- **嵌入服务**：OpenAI-compat API，带 LRU 缓存（8192 条目）+ 熔断器（5 次失败 / 5 分钟冷却）

## 二、写入机制（Remember）

### 2.1 五条写入路径

```
┌─────────────────────────────────────────────────────────────────┐
│  ① KeyInfoExtractor 正则提取（实时，用户消息中）                  │
│     → URL / 账号 / 密码 / 目标 / 截止日期 / 记住指令 / 身份声明   │
│     → 置信度 0.82，直接 remember()                                │
│                                                                  │
│  ② LLMFactExtractor 后台提取（每轮结束后，fire-and-forget）       │
│     → 3-8s 延迟，不阻塞 turn 返回                                 │
│     → Phase 8 可选 Mem0 风格决策（ADD/UPDATE/DELETE/NOOP）        │
│                                                                  │
│  ③ Agent 工具调用（显式）                                         │
│     → memory(action='remember'/'forget'/'correct'/'search')      │
│                                                                  │
│  ④ UI 手动操作（记忆面板）                                         │
│     → 直接调用 MemoryService.remember()                           │
│                                                                  │
│  ⑤ Persona 文件编辑（IDENTITY.md / USER.md 等）                  │
│     → upsert_persona_manual() 同步到 V2                           │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 写入流程（remember() 内部）

```
text + kind + scope
    ↓
[1] 计算确定性 ID: kind:scope:hash12(text)
    ↓
[2] 嵌入文本（EmbeddingService，LRU 缓存 + 3 次重试 + 熔断器）
    ↓
[3] 近重复检测（KNN top-3，cosine distance < 0.15 → 合并为证据投票）
    ↓
[4] 关系扫描（KNN top-10，SAME_TOPIC / CONTRADICTS / 共享实体桥接）
    ↓
[5] 自动提升层（evidence_count ≥ 3 → working → long_term）
    ↓
[6] upsert 到 VectorBackend + 写 GraphBackend 边
    ↓
[7] 注册到 EntityStore（Wave-32+，O(1) 共享实体查询）
```

### 2.3 关键阈值

| 阈值 | 值 | 含义 |
|------|-----|------|
| `NEAR_DUPLICATE_DISTANCE_THRESHOLD` | 0.15 | 近重复合并（不同措辞的同一事实） |
| `SAME_TOPIC_DISTANCE_THRESHOLD` | 0.30 | 同主题关联（默认边类型） |
| `CONTRADICTS_DISTANCE_THRESHOLD` | 0.25 | 矛盾检测（仅 kind=correction 时触发） |
| `LONG_TERM_PROMOTE_THRESHOLD` | 3 | 证据投票数达到后自动晋升 long_term |

## 三、召回机制（Recall）

### 3.1 五条召回路径

```
┌─────────────────────────────────────────────────────────────────┐
│  ① Structural axis（结构轴）— 始终注入 system prompt              │
│     → bucket 路由: user_identity → USER.md                      │
│     → bucket 路由: agent_identity → IDENTITY.md                 │
│     → bucket 路由: user_preference → USER.md                    │
│     → 由 v2_renderer 在 daemon 启动时渲染为 .md 文件            │
│                                                                  │
│  ② Similarity axis（相似轴）— 每轮用户消息触发                     │
│     → auto_recall.py: recall_for_message()                      │
│     → 嵌入用户消息 → KNN top-k → prepend <recalled> block        │
│     → 默认 k=8, min_similarity=0.65, timeout=1.0s               │
│                                                                  │
│  ③ render_for_prompt() — 四段式上下文渲染                         │
│     → 用户档案（preference/identity/correction，user scope）       │
│     → 项目档案（project/commitment，project scope）                │
│     → 决定记录（decision，全 scope）                              │
│     → 与本轮相关（向量召回 top-k，2026-06-06 加 distance≤0.40 过滤）│
│                                                                  │
│  ④ Graph proactive_recall — 1-hop 邻居扩展                       │
│     → CONTRADICTS / SUPERSEDES / SAME_TOPIC 边遍历              │
│     → 在 recall() 中自动 enrich 到 related_relations              │
│                                                                  │
│  ⑤ Autobiographical summary — 长会话摘要                         │
│     → 跨多 turn 的叙事连续性，由 autobiographical_memory.py 管理   │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 三因子排序（Phase 8 ⑦，Generative Agents 风格）

```python
score = 1.0 * relevance(cosine) + 1.0 * recency(exp_decay, 7天半衰期) + 1.0 * importance(confidence)
```

- **relevance**: 查询向量与事实嵌入的 cosine 相似度
- **recency**: 基于 `ts_last` 的指数衰减，7 天半衰期
- **importance**: 事实自身的 confidence（随 evidence_count 增长）

### 3.3 强化机制（Phase 8 ⑪，MemoryBank 效应）

当事实被实际注入 prompt 后，自动 bump `ts_last`（最小间隔 30 分钟），使频繁有用的记忆保持"新鲜"。

## 四、策展与维护（Curator）

### 4.1 四阶段策展流程

```
┌─────────────┐    ┌─────────────┐    ┌─────────────────┐    ┌─────────────┐
│  ① dedup    │ → │  ② prune    │ → │  ③ contradict   │ → │ ④ crystallize│
│  向量聚类    │    │  降权低价值  │    │  LLM 矛盾检测    │    │ LLM 语义结晶  │
│  cosine≥0.86│    │  30天+单证据 │    │  temporal失效   │    │ 碎片→规范表述 │
│  时间预算制  │    │  confidence≤0.55 → floor 0.15 │    │ 保守策略      │
└─────────────┘    └─────────────┘    └─────────────────┘    └─────────────┘
```

### 4.2 调度机制

- **旧机制**：每 24 次 sweep 触发（开发环境 daemon 每 30 分钟重启 → 从未触发）
- **新机制**：墙钟时间持久化（`last_curate_ts` 写入磁盘），重启后仍能判断"是否该整理了"

### 4.3 维护扫描限制

```python
_MAINTENANCE_SCAN_LIMIT = 5000  # dedup / backfill / sweep 等操作的扫描上限
```

超过 5000 条事实时，维护操作只能看到部分数据，会触发 `_maybe_warn_scan_truncated` 日志警告，但**不会自动分页**。

## 五、已识别问题清单

### 🔴 高优先级

#### M-1: 召回超时过短（1.0s）导致大存储下频繁空返回

**位置**: `xmclaw/daemon/auto_recall.py:42`  
**代码**:
```python
_DEFAULT_TIMEOUT_S = 1.0   # 2026-05-29 事故后加的紧急上限
```

**问题**: 在 LanceDB 存储超过数千条事实时，嵌入 + KNN 搜索可能超过 1.0s。超时后 `recall_for_message` 返回 `[]`，用户轮次**完全无记忆注入**。测试 `test_recall_returns_empty_on_timeout` 明确验证了这一行为。

**影响**: 大存储用户会体验到"记忆突然失效"——不是记忆丢了，是召回被掐断了。

**建议**: 
- 将默认超时提升至 3.0s（嵌入缓存命中时通常 < 500ms，冷启动时 1-2s）
- 或改为分层超时：嵌入 2.0s + 搜索 2.0s，总预算 4.0s
- 添加 metrics：记录 `recall_timeout_count` 供监控

---

#### M-2: Hybrid 召回（BM25）默认关闭，CJK 关键词查询效果差

**位置**: `xmclaw/daemon/auto_recall.py:71`  
**代码**:
```python
async def recall_for_message(..., use_hybrid: bool = False, ...)
```

**问题**: 纯向量召回对中文关键词（如"FastAPI"、"网店"）的匹配质量依赖嵌入模型的语义理解。当用户查询包含特定术语或罕见标识符时，cosine 相似度可能无法准确反映相关性。

**现状**: `recall_hybrid()` 已实现（向量 + BM25 融合，权重 0.6/0.4），但默认关闭。BM25 路径需要 `rank_bm25` 包，且每次查询重建 Python BM25 索引（corpus_cap=500，bm25_deadline_s=0.5）。

**建议**:
- 默认启用 hybrid（当 `rank_bm25` 可用时）
- 或添加配置项 `memory_auto_recall_hybrid: bool = True`
- 长期：等待 LanceDB 原生 FTS（Phase 5）

---

#### M-3: 无来源标签（provenance），无法区分用户确认 vs 模型推断

**位置**: `xmclaw/memory/v2/models.py:Fact`  
**现状**: `Fact` 模型有 `source_event_id`（指向 events.db），但**没有 provenance 字段**标识写入来源：

| 来源 | 当前标识方式 | 问题 |
|------|-------------|------|
| KeyInfoExtractor 正则 | 无 | 无法区分"用户亲口说"vs"模型推断" |
| LLMFactExtractor 后台 | `reason: "llm_auto_extract:add"` | 仅在事件日志中，不在 Fact 上 |
| Agent 工具调用 | 无 | 无法区分显式记忆 vs 自动提取 |
| UI 手动 | 无 | 无法区分 |
| Persona 文件 | `kind=persona_manual` | 仅此类有标识 |

**影响**: 策展时无法优先保留用户确认的事实；调试时无法追溯"这条记忆从哪来的"。

**建议**: 在 `Fact` 模型添加 `provenance: str` 字段（如 `"user_confirmed"`, `"auto_extract_regex"`, `"auto_extract_llm"`, `"tool_invoked"`, `"manual_ui"`）。

---

#### M-4: 跨会话长任务工作流断裂

**位置**: `xmclaw/daemon/session_store.py` + `xmclaw/daemon/hop_loop.py`  
**现状**: `SessionStore` 保存对话历史（`Message` 列表），但：

1. **无任务状态机**: 长任务（如"帮我写一份 50 页报告"）分 10 个会话完成时，每个新会话从零开始，不知道之前完成了哪几章
2. **无工作流记忆**: 没有"当前进行中的任务 + 已完成阶段 + 待办清单"的结构化存储
3. **session 间 recall 盲区**: `render_for_prompt` 的"与本轮相关"只召回当前查询相关的片段，不会主动注入"你正在进行的任务上下文"

**对比**: OpenClaw 的 `session_store` 扩展了 `task_context` 字段，新会话自动继承。

**建议**:
- 引入 `TaskContext` 模型（task_id, goal, completed_steps, pending_steps, last_session_id）
- 在 `render_for_prompt` 中添加"进行中任务"小节
- 或利用现有的 `project` scope + `commitment` kind 存储任务状态

---

#### M-5: 策展人 contradict / crystallize 阶段 LLM 成本过高

**位置**: `xmclaw/memory/v2/curator.py:219-256`  
**现状**: 
- `contradict` 阶段：全量扫描 scope 内事实，LLM 判断矛盾对
- `crystallize` 阶段：全量扫描，LLM 合成规范表述
- 每次策展（默认每天一次）都触发 LLM 调用，**无论事实是否有变化**

**成本估算**: 假设 500 条事实，batch_size=60，crystallize 需要 ~8 次 LLM 调用；contradict 需要 ~N²/2 次比较（实际通过 LLM 批量处理）。

**建议**:
- 增量策展：只扫描 `ts_last > last_curate_ts` 的新事实/变更事实
- 或添加 "事实变化检测"：比较上次策展后的新增数量，< 10 条时跳过 LLM 阶段
- 将 contradict/crystallize 改为按需触发（用户报告"记忆混乱"时）而非每日自动

### 🟡 中优先级

#### M-6: 无跨实例记忆同步

**现状**: LanceDB 是本地文件（`~/.local/share/xmclaw/memory/`），无云同步。用户换机器或重装后记忆丢失。

**对比**: OpenClaw 提供 `memory-lancedb` cloud sync。

**建议**: 添加可选的同步后端（如 SQLite → S3 / WebDAV / 自建同步服务），或导出/导入 CLI。

---

#### M-7: 无原生 FTS5 全文搜索

**现状**: 关键词搜索依赖：
1. `recall()` 的 `keyword_only=True` 路径 → LanceDB `LIKE` 查询（非 FTS）
2. `recall_hybrid()` 的 BM25 → Python `rank_bm25` 库（每次查询重建索引）

**问题**: 无原生 FTS5 意味着：
- 中文分词质量差（BM25 默认按空格分词）
- 大存储下 BM25 构建耗时（500 条 corpus 虽 capped，但仍需 100-500ms）
- 无法支持前缀匹配、模糊搜索、同义词扩展

**建议**: 长期等待 LanceDB 原生 FTS；短期可集成 `jieba` 中文分词 + 预构建 BM25 索引（后台更新）。

---

#### M-8: LanceDB 损坏检测过于激进

**位置**: `xmclaw/memory/v2/backend_lancedb.py`  
**现状**: `_corrupted` 标志一旦触发（任何 lance error），**永久禁用 V2 直到手动修复**。

```python
if getattr(_vec, "_corrupted", False):
    return ""  # render_for_prompt 直接短路
```

**问题**: 瞬态错误（如磁盘满、文件锁）也会触发永久禁用，用户需手动删除 `memory.db` 重建。

**建议**: 区分"可恢复错误"（重试 3 次）vs "永久性损坏"（需重建）；添加自动修复尝试（如 `lance.dataset.cleanup()`）。

---

#### M-9: 维护扫描截断导致大存储下策展不完整

**位置**: `xmclaw/memory/v2/service.py:79`  
**代码**:
```python
_MAINTENANCE_SCAN_LIMIT = 5000
```

**问题**: dedup / backfill / sweep 等操作只扫描前 5000 条事实。当存储超过 5000 条时，**旧事实永远不被策展**。

**现状**: 已有 `_maybe_warn_scan_truncated` 日志警告，但无自动分页。

**建议**: 实现 cursor-based 分页扫描（按 `ts_last` 或 `id` 排序分批处理）。

---

#### M-10: 2026-06-06 上下文污染修复的阈值合理性待验证

**位置**: `xmclaw/memory/v2/service.py:2618`  
**代码**:
```python
new_hits = _rank([
    h for h in relevant_hits
    if h.fact.id not in seen_ids
    and float(getattr(h, "distance", 0.0) or 0.0) <= 0.40
])
```

**背景**: 紧急修复——对"需要"这类无真正相关记忆的输入，top-k 召回会塞入 8 条最近邻，污染上下文。

**问题**: 阈值 0.40（cosine similarity ≈ 0.6）是经验值，未经 A/B 测试验证。可能过于严格导致"该召回的没召回"，或过于宽松导致"不该召回的还召回"。

**建议**: 添加配置项 `memory_recall_distance_threshold`（默认 0.40），允许用户调整；收集 metrics 分析阈值分布。
